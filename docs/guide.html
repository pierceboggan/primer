Getting Started

Set up Primer to analyze, generate, and optimize AI-readiness for your repositories.

Prerequisites

RequirementDetails
Node.js 20+ Runtime environment. nodejs.org
GitHub Copilot CLI Required for instruction generation. Verify with copilot --version
Copilot Auth Run copilot and use /login to authenticate
GitHub CLI (optional) For batch and PR workflows. brew install gh && gh auth login
Azure DevOps PAT (optional) Set AZURE_DEVOPS_PAT env var for Azure workflows

Installation

Run directly (no install)

$ npx github:pierceboggan/primer <command>

Install globally

$ npm install -g github:pierceboggan/primer
$ primer --version

Clone and build from source

$ git clone https://github.com/pierceboggan/primer.git
$ cd primer
$ npm install && npm run build
$ npm link

VS Code Extension

Install from the VS Code Marketplace, or search for "Primer" in the Extensions view (Ctrl+Shift+X).

Quick Start

The fastest way to get started is with primer init. It runs an interactive setup that analyzes your repo, generates AI instructions, and creates config files: 

$ cd your-project
$ primer init

This will:

  1. Detect your languages, frameworks, and project structure
  2. Generate a copilot-instructions.md tailored to your codebase
  3. Create .vscode/mcp.json with relevant MCP server configs
  4. Set up .vscode/settings.json optimized for your stack

Use --yes to accept all defaults without prompts, or --force to overwrite existing files.

Core Workflow

Primer's five core capabilities can be used independently or as a progressive workflow. Here's the recommended order for setting up a repository:

1. Analyze your repository

Start by understanding your repo's structure. The analyze command detects languages, frameworks, package managers, and monorepo workspaces: 

$ primer analyze
$ primer analyze --output report.md

The analysis results feed into instruction generation and readiness scoring. In VS Code, results populate the Analysis sidebar view.

2. Generate AI instructions

Generate copilot-instructions.md or AGENTS.md files using the Copilot SDK. Instructions are analyzed from your actual code, not templated: 

# Generate root instructions
$ primer instructions

# Generate per-area instructions for monorepos
$ primer instructions --areas

# Generate just one specific area
$ primer instructions --area frontend

3. Generate configurations

Create MCP server configs and VS Code settings tailored to your tech stack: 

$ primer generate mcp      # .vscode/mcp.json
$ primer generate vscode   # .vscode/settings.json

4. Check AI readiness

Score your repository across 9 maturity pillars and get a readiness level from 1–5: 

$ primer readiness
$ primer readiness --visual --output readiness.html

The 9 pillars span two groups:

  • Repo Health — Style/linting, build system, testing, documentation, dev environment, code quality, observability, security/governance
  • AI Setup — Instructions, MCP servers, agents, skills

5. Evaluate effectiveness

Measure whether your instructions actually improve AI responses by running evaluations with a judge model: 

# Create starter eval config
$ primer eval --init

# Run evaluations
$ primer eval

The eval compares AI responses with and without custom instructions, then a judge model scores which response better follows codebase conventions.

Monorepo Support

Primer automatically detects monorepo structures (npm/yarn/pnpm workspaces, Lerna, Nx, Rush, etc.) and maps logical areas like frontend, backend, infrastructure.

For monorepos, you can generate per-area instructions scoped with applyTo globs: 

# Generate instructions for all detected areas
$ primer instructions --areas

# Generate for a specific area
$ primer instructions --area api

# Generate per-app configs
$ primer generate instructions --per-app

Per-area readiness breakdowns are also available: 

$ primer readiness --per-area

Readiness Policies

Policies are composable JSON files that customize which readiness criteria are evaluated, override scoring metadata, and set pass-rate thresholds. Chain multiple policies or distribute them org-wide. 

# Apply a single policy
$ primer readiness --policy ./policies/strict.json

# Chain multiple policies
$ primer readiness --policy ./base.json,./team-overrides.json

Policy format

{
  "name": "my-org-policy",
  "criteria": {
    "disable": ["lint-config"],
    "override": {
      "readme": { "impact": "high", "level": 2 }
    }
  },
  "extras": {
    "disable": ["pre-commit"]
  },
  "thresholds": {
    "passRate": 0.9
  }
}

See the readiness command reference for all policy options.

Batch Operations

Process multiple repositories across GitHub organizations or Azure DevOps projects. Primer supports both interactive TUI and headless modes.

GitHub — Interactive

# Launch the TUI to select repos from your org
$ primer batch

GitHub — Headless

$ primer batch owner/repo1 owner/repo2 --json
$ echo "owner/repo1\nowner/repo2" | primer batch --json

Azure DevOps

$ export AZURE_DEVOPS_PAT="your-pat-here"
$ primer batch org/project/repo --provider azure

Batch readiness report

$ primer batch-readiness --output report.html

CI/CD Integration

Use Primer's --fail-level flag to enforce readiness and evaluation thresholds in your CI pipeline.

Readiness gate

Exit with code 1 if the readiness level falls below a threshold: 

# Fail if readiness level < 3
$ primer readiness --fail-level 3 --json

Eval gate

Exit with code 1 if the eval pass rate drops below a threshold: 

# Fail if pass rate < 80%
$ primer eval --fail-level 80 --json

Example GitHub Actions workflow

- name: Check AI Readiness
  run: npx github:pierceboggan/primer readiness --fail-level 3 --json

- name: Run Eval
  run: npx github:pierceboggan/primer eval --fail-level 80 --json
Tip
Always use --json in CI to get machine-readable output on stdout. Human-readable progress goes to stderr and won't interfere with JSON parsing.
